# Настройка входа (Google и Яндекс)

В проекте используется [NextAuth.js v4](https://next-auth.js.org/) с маршрутом `pages/api/auth/[...nextauth].js`. Данные библиотек по-прежнему хранятся **в браузере** (`localStorage`); вход нужен, чтобы включить продуктовое правило «не более трёх библиотек» и в будущем привязать аккаунт к облаку.

## 1. Переменные окружения

Скопируйте `.env.example` в `.env.local` и заполните:


| Переменная                                  | Назначение                                                                       |
| ------------------------------------------- | -------------------------------------------------------------------------------- |
| `NEXTAUTH_URL`                              | Публичный URL сайта, например `http://localhost:3000` или `https://ваш-домен.ru` |
| `NEXTAUTH_SECRET`                           | Случайная строка для подписи сессии. Сгенерировать: `openssl rand -base64 32`    |
| `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` | OAuth 2.0 клиент Google                                                          |
| `YANDEX_CLIENT_ID` / `YANDEX_CLIENT_SECRET` | OAuth-приложение Яндекса                                                         |


Провайдер подключается **только если** заданы оба ключа для этого провайдера. Можно включить только Google или только Яндекс.

## 2. Google Cloud Console

1. [Google Cloud Console](https://console.cloud.google.com/) → проект → **APIs & Services** → **Credentials** → **Create credentials** → **OAuth client ID**.
2. Тип приложения: **Web application**.
3. **Authorized redirect URIs** добавьте:
  - локально: `http://localhost:3000/api/auth/callback/google`
  - продакшен: `https://ваш-домен.ru/api/auth/callback/google`
4. Скопируйте Client ID и Client Secret в `.env.local`.

## 3. Яндекс OAuth

1. Откройте [создание приложения OAuth](https://oauth.yandex.ru/client/new) (Яндекс ID для разработчиков).
2. Укажите **Redirect URI**:
  - `http://localhost:3000/api/auth/callback/yandex`
  - `https://ваш-домен.ru/api/auth/callback/yandex`
3. Запросите права **login:info** и **login:email** (как в коде провайдера).
4. Подставьте ID и пароль приложения в `YANDEX_CLIENT_ID` и `YANDEX_CLIENT_SECRET`.

## 4. Проверка

1. Заполните `.env.local`, перезапустите dev-сервер.
2. Откройте главную → иконка профиля внизу слева (третья кнопка) или `/auth/signin`.
3. После успешного входа создание библиотеки доступно при **менее чем трёх** списках в `localStorage`.

## 5. Частые ошибки

- **Redirect_uri mismatch** — в консоли Google/Яндекса должен совпадать URL **точно** с тем, что открывает NextAuth (`/api/auth/callback/...`).
- `**NEXTAUTH_URL` не совпадает с реальным доменом** — сессии и редиректы OAuth ломаются; на проде задайте реальный HTTPS-URL.
- **Нет провайдеров** — если не задан ни один пара ключей, страница входа откроется, но кнопки провайдера не сработают (в консоли сервера NextAuth предупредит).

## 6. Дальнейшая доработка

Сейчас лимит «3 библиотеки» проверяется на клиенте. Для облачного хранения понадобятся свои API и привязка `user.id` из сессии к записям в БД — тогда лимит нужно дублировать на сервере.  
  
  
  
**Юридические риски (общая оценка, не юридическая консультация)**

То, что вы **не продаёте шрифты**, убирает целый пласт вопросов (коммерческая лицензия на отдельные коммерческие шрифты, договоры купли-продажи и т.п.). Но это **не обнуляет** все риски:

- **Персональные данные.** Вход через Google/Яндекс даёт вам доступ к идентификаторам, иногда email и имени. Это обработка ПДн: нужны **политика конфиденциальности**, правовое основание, в РФ — ориентир на **152‑ФЗ** и типовые требования к согласию/уведомлению; при аудитории из ЕС/ЕЭЗ — ориентир на **GDPR** (юрисдикция зависит от того, кого вы целенаправленно обслуживаете и как устроен хостинг).
- **Пользовательский контент.** Списки библиотек — это пользовательские наборы; при облачной синхронизации позже появятся вопросы хранения, удаления, передачи третьим лицам — это снова про политику и договор с пользователем.
- **Шрифты и права третьих лиц.** Вы не «продаёте шрифт», но если когда‑нибудь будете **раздавать файлы шрифтов** или крупные выгрузки с чужих каталогов, остаются обязанности по **лицензиям** (например, OFL для Google Fonts) и по **условиям API/каталогов**. Сейчас при локальном хранении списков метаданных риски ниже, чем у маркетплейса файлов, но юристу всё равно стоит посмотреть формулировки в **Terms** и описание продукта.
- **Платёж «сверху» в будущем.** Появление оплаты добавляет **закон о защите прав потребителей**, правила платёжных посредников, НДС/чеки — это отдельный слой, к нему готовятся заранее.

Итого: отсутствие продажи шрифтов **снижает** коммерческие лицензионные риски «как у магазина шрифтов», но **не заменяет** работу по ПДн и пользовательским условиям. Для уверенности по РФ/международке имеет смысл **одна консультация юриста**, когда зафиксируете: где хостинг, есть ли платежи, только РФ или нет.
